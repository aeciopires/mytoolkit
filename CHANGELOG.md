## [1.1.0] - 2026-07-20

### Added

- MCP (Model Context Protocol) server: a 4th surface alongside the web UI, REST API, and CLI, exposing all 15 tools as 16 MCP tools (JWT splits into `jwt_encode`/`jwt_decode`) via the new `mytoolkit mcp` subcommand. Built on the official `github.com/modelcontextprotocol/go-sdk`, it calls the exact same `internal/tools/<name>` pure functions the other three surfaces already call ŌĆö zero duplicated business logic. Lives at `app/internal/mcp/` (one file per tool, self-registering via `init()`, mirroring `internal/cli/<name>.go`'s pattern) plus `app/internal/cli/mcp.go` for the cobra wiring; `mcp/README.md` at the repo root documents installation and client configuration (Claude Desktop, Claude Code), with example config files under `mcp/examples/` and a Mermaid workflow diagram.
- Two transports: `stdio` (default, for local MCP clients) and streamable `http` (`--transport http --port 8081`, for a shared/remote MCP endpoint), configurable via `MYTOOLKIT_MCP_TRANSPORT`/`MYTOOLKIT_MCP_PORT`/`--transport`/`--port` following the same flag > env > default precedence as every other MyToolkit setting.
- `qrcode`'s MCP tool returns an `image/png` content block (base64 on the wire) rather than JSON text, and `k8s-validate`'s MCP tool matches the REST endpoint's behavior of treating a parseable-but-semantically-invalid document as a successful call (`isError: false`) whose structured result reports `"valid": false` ŌĆö both documented exceptions, not oversights.
- `helm/mytoolkit`: optional `mcp.enabled` Deployment/Service pair (same image, `args: ["mcp", "--transport", "http", ...]`, TCP-socket liveness/readiness probes since the MCP endpoint isn't a plain HTTP GET target), plus its own dedicated ConfigMap, and optional HorizontalPodAutoscaler (`mcp.autoscaling.enabled`), PodDisruptionBudget (`mcp.pdb.enabled`), ServiceAccount (`mcp.serviceAccount.create`, falls back to the shared ServiceAccount when disabled), Secret (`mcp.secret.enabled`), and Ingress (`mcp.ingress.enabled`) ŌĆö mirroring the equivalent toggles already available for the main Deployment. `docker-compose.yml`: optional `mytoolkit-mcp` service under the `mcp` Compose profile. `Makefile`: `mcp-run`/`mcp-run-http` targets.
- Verified end-to-end against the real built binary: a throwaway MCP client driving `mytoolkit mcp` over both a real `os/exec` stdio subprocess and a `curl` streamable-HTTP handshake ŌĆö `tools/list` (16 tools), a `base64` round trip, an intentionally invalid option producing `isError: true`, `k8s-validate`'s non-error semantic-invalid result, and `qrcode`'s image content block all confirmed against actual output, not hand-typed ŌĆö plus 30 new Go tests under `app/internal/mcp/`.
- MCP Prometheus metrics: `mytoolkit_mcp_requests_total{method,status}`/`mytoolkit_mcp_request_duration_seconds{method}` for every JSON-RPC method, `mytoolkit_mcp_tool_calls_total{tool,status}`/`mytoolkit_mcp_tool_call_duration_seconds{tool}` for `tools/call` specifically, and `mytoolkit_mcp_sessions_total` ŌĆö recorded by one `server.AddReceivingMiddleware` hook (`app/internal/mcp/metrics.go`) covering all 16 tools and every JSON-RPC method, no per-handler instrumentation. Kept as a metric family separate from `mytoolkit_http_*`/`mytoolkit_tool_usage_total` (distinct client population). `GET /metrics` is mounted alongside the MCP JSON-RPC endpoint on the same `http.ServeMux` when `--transport http` ŌĆö `stdio` has no listening port, so it isn't exposed there (inherent to the transport, not a gap).
- Grafana dashboard: a new "MCP Server" row (12 panels ŌĆö tool call rate/error-rate/latency percentiles by tool, request rate by JSON-RPC method, session count, current-totals table, and a tool-call ranking) added to `observability/mytoolkit-dashboard.json`. `observability/prometheus.yml` gained a `mytoolkit-mcp` scrape job (the optional Compose `mcp` profile service); `helm/mytoolkit/templates/mcp-deployment.yaml` gained `prometheus.io/scrape` annotations and a new optional `mcp-servicemonitor.yaml`. Verified end-to-end against the real docker-compose stack: real `initialize`/`tools/call` JSON-RPC traffic against `mytoolkit-mcp`, confirmed present in Prometheus via direct API query, and confirmed rendering in the provisioned Grafana dashboard via Grafana's own datasource-proxy API (not just visual inspection of the JSON).

## [1.0.0] - 2026-07-10

### Added

- Initial implementation of MyToolkit: a Go web application and CLI exposing 15 tools.
- REST API under `/api/v1/tools/<slug>` for every tool, with a shared JSON success/error envelope.
- CLI subcommand per tool, with a shared `--in`/`--out` (file-or-stdio) convention.
- `mytoolkit --version`/`-v`, reading from the repo-root `VERSION` file ŌĆö the single source of truth also used to tag Docker images.
- Server-rendered web UI with a Material Design 3ŌĆōinspired dark/light theme; pages accept a `?theme=light|dark` override for deterministic screenshots/demos.
- `/healthz`, `/readyz`, `/metrics` (Prometheus), and `/api/v1/metrics/ranking` (usage ranking) endpoints.
- Structured JSON logging via zerolog, with request correlation IDs.
- Multi-stage, multi-arch (linux/amd64, linux/arm64) Dockerfile, validated with `docker buildx`, and `docker-compose.yml`.
- `make docker-push`: interactive Docker Hub publish (hidden credential prompt, `--password-stdin`, never logged).
- Helm chart under `helm/mytoolkit` ŌĆö Prometheus scrape annotations, `helm-docs`-generated chart README, deployed and `helm test`-verified against a real `kind-multinodes` cluster.
- Unit tests for every tool's business logic, plus per-feature documentation (`docs/api|cli|testing`, with a Mermaid workflow diagram each) and `.skills/`.
- JWT Encode/Decode: multi-algorithm support via `golang-jwt/jwt/v5` (already a dependency, previously only used for HMAC). Adds RSA (`RS256`/`RS384`/`RS512`), RSA-PSS (`PS256`/`PS384`/`PS512`), ECDSA (`ES256`/`ES384`/`ES512`), and EdDSA alongside the original HMAC algorithms ŌĆö 13 total ŌĆö selectable via a new web UI combobox, plus the existing CLI `--algorithm` flag and REST `options.algorithm`. `HS256` remains the default, unchanged. Asymmetric algorithms sign/verify with a new `key` parameter (PEM-encoded ŌĆö private key to encode, public key to verify a decode; CLI: `--key <file|->`; REST/web: `options.key`), independent of the existing HMAC-only `secret`. Verification derives which of `secret`/`key` applies from the token's own `alg` header, not a caller hint, so pasting the wrong one fails verification cleanly (`valid: false`) instead of erroring or silently succeeding ŌĆö verified directly, including a full round trip through the CLI, REST, and the actual running web page against a real `openssl`-generated RSA key pair.
- Shared navigation shell on every page: a hamburger menu opening a Material Design 3 "modal navigation drawer" listing all tools (with a scrim, closable via scrim click/Ō£Ģ/Escape); a client-side search bar (`#tool-search`) that filters tools live by name and description with zero network calls; a "ŌåÉ Back to Home" button on every tool page; and a footer (developer/contact links) on every page.
- M3 component pass across the web UI, following `m3.material.io`'s spacing, buttons, icon-buttons, checkbox, lists, navigation-bar, navigation-drawer, radio-button, search, switch, and text-fields guidance: a 4dp spacing scale and state-layer opacity tokens in `theme.css`, restyled checkboxes and a new switch component (single on/off settings like "Decode" now use a switch instead of a checkbox), consistent text-field focus rings, and shared list-item styling reused by both the nav drawer and search results.
- `internal/registry/registry_test.go`: a regression test (`TestToolJSONFieldsAreLowercase`) locking in `registry.Tool`'s JSON contract.
- `internal/web/static/icons/favicon.svg`, referenced via `<link rel="icon">`.
- JSON Tree Viewer overhaul: error messages now report the exact 1-indexed line and column of the problem (e.g. `invalid character '}' looking for beginning of value (at line 4, column 1)`), not just a bare parser message; trailing content after a complete JSON value (e.g. `{"a":1}garbage`, or two concatenated values) is now rejected instead of silently ignored; the web page gained an explicit "Generate Tree View" action (conversion no longer happens on every keystroke ŌĆö large pasted API responses made that wasteful and janky), plus "Expand All"/"Collapse All"/"Clear" buttons; tree nodes are now color-coded by value type (VS CodeŌĆōinspired: keys, strings, numbers, booleans, and null each get a distinct, theme-aware color) and show `Object {N keys}`/`Array [N items]` summaries instead of a generic `type(size)` placeholder.
- JSON Formatter web page overhaul: now runs entirely client-side via the browser's native `JSON.parse()`/`JSON.stringify()` (no `fetch()` call from the interactive tool), so validation results and error messages exactly match real Node.js/browser behavior instead of Go's `encoding/json` wording. New, explicit actions replace live-as-you-type formatting: "Validate JSON" (green success banner on valid input, red error banner with the exact `JSON.parse()` message on invalid input, output untouched either way), "Beautify" (2-space indent), "Minify", "Copy Output", "Clear". Adds a new `.success-banner` style (light/dark) alongside the existing `.error-banner`. REST (`POST /api/v1/tools/json-format`) and CLI (`mytoolkit json-format`) are unchanged, still full Go implementations for scripted use; as with JSON to TOON Converter, `mytoolkit_tool_usage_total{tool="json-format"}` no longer reflects web page usage, only REST/CLI.
- Character, Word & Line Counter: added a "Clear" button that empties the input and resets all four stats to `0` (previously required manually selecting and deleting the textarea contents).
- YAML Formatter overhaul, driven by a re-read of the YAML spec (yaml.org/spec/1.2.2): (1) **fixed a real spec-compliance bug** ŌĆö multi-document streams (`---`-separated) previously had every document after the first silently discarded; `Format` now decodes and reformats the entire stream, and the encoder re-inserts `---` between documents automatically. (2) Added a `style` option (`block`, default, or `flow`) that forces consistent collection style across the whole document ŌĆö the YAML equivalent of JSON Formatter's pretty/minify, and always lossless per the spec ("node style is a presentation detail... not reflected in the representation graph"). Previously the formatter only re-indented and left flow-style input (`{a: 1}`) untouched, so mixed-style input never actually got consistent formatting. New `--style` CLI flag and web "Style" select. A whitespace/comment-only input (which decodes to zero YAML documents without a parse error) is now treated as empty input rather than silently returning an empty result. Verified and documented actual `yaml.v3` behavior for comments, anchors/aliases, merge keys, tab-indentation rejection, and the "Norway problem" (`yes`/`no`/`NO`) ŌĆö all preserved correctly or already spec-correctly rejected; see `.skills/yaml-format/SKILL.md`.
- Grafana dashboard for the `prometheus`/`grafana` `docker-compose.yml` services: `observability/mytoolkit-dashboard.json` covers every metric the app exposes ŌĆö HTTP request rate/error-rate/latency percentiles by tool and status (`mytoolkit_http_requests_total`, `mytoolkit_http_request_duration_seconds`), per-tool usage (`mytoolkit_tool_usage_total`, the same data behind `GET /api/v1/metrics/ranking`), and Go runtime/process health (goroutines, memory, GC, CPU, open FDs, network I/O). Auto-provisioned via `observability/grafana/provisioning/{datasources,dashboards}/*.yml` ŌĆö `docker compose up --build` now gives a working dashboard with zero manual Grafana setup. Every query was verified against a real running binary's `/metrics` output, and the whole stack was verified end-to-end (real traffic, real Grafana login, screenshotted rendering) ŌĆö see `.skills/observability/SKILL.md`.
- Interactive Swagger/OpenAPI documentation at `GET /swagger/index.html` (`swaggo/swag` + `swaggo/http-swagger`), covering all 19 JSON REST endpoints: every `/api/v1/tools/<slug>` tool endpoint, `GET /api/v1/tools`, `GET /api/v1/metrics/ranking`, `/healthz`, `/readyz`. Generated from `@`-comment annotations via a new `make swagger-gen` target (pinned `go run github.com/swaggo/swag/cmd/swag@v1.16.6`, no separate install needed) into `app/docs/` (committed, like the Helm chart's generated README). Each of the 9 tools wired via the generic `handlers.Wrap(...)` got a small named wrapper function (e.g. `base64Handler()`) purely to carry its annotation, since `swag` needs a real function declaration to attach a doc comment to. New `internal/httpapi/swagger_test.go` regression test asserts every tool in `registry.All()` has a corresponding Swagger path, so a future tool added without annotations fails a test instead of silently missing from the docs. `/metrics` (Prometheus text format) is deliberately not included ŌĆö it isn't JSON/OpenAPI-shaped. Verified end-to-end: a real running binary's `/swagger/doc.json` and `/swagger/index.html`, and a Playwright-driven "Try it out" execution against the live `base64` endpoint returning a real signed response. See `.skills/swagger/SKILL.md`.
- `make helm-set-appversion`: syncs `helm/mytoolkit/Chart.yaml`'s `appVersion` field to the root `VERSION` file (portable `sed -i.bak` + `rm`, works on both GNU and BSD/macOS `sed`). Now a prerequisite of `make helm-docs`, so every chart-docs regeneration keeps `appVersion` from silently drifting out of sync with `mytoolkit --version`/the Docker image tag. The chart's own `version:` field (semver of the chart itself) is untouched.

### Changed

- Go/HTML/CSS/JS source (`cmd/`, `internal/`) moved under `src/` (its own Go module root), separating application source from planning, documentation, and deployment files at the repo root.

### Fixed

- `README.md` previously listed a non-existent "SHA-1024" hash algorithm; the Hash Generator implements MD5, SHA-1, SHA-256, and SHA-512.
- The Helm chart's test hook was initially placed at `tests/test-connection.yaml`; Helm only discovers hooks under `templates/`, so it was silently never run until moved to `templates/tests/test-connection.yaml`.
- Verification pass across all 12 tools: added missing Mermaid workflow diagrams to 7 `docs/api/<tool>.md` files that didn't have one (base64, case-convert, hash-gen, jwt, qrcode, text-count, url-encode).
- `docs/api/url-encode.md` documented a nonexistent `options.mode` field; the real field is `options.decode` (boolean). Fixed to match `urlencode.Options`.
- `docs/cli/url-encode.md`'s example used `echo` (which appends a trailing newline that gets percent-encoded as a stray `%0A`); switched to `echo -n` and documented why.
- `docs/cli/json-tree.md`'s example output used a hand-typed compact JSON format that didn't match the real `json.MarshalIndent` output; corrected to the actual (fully expanded) output.
- `docs/api/json-format.md` and `docs/api/yaml-format.md` error examples used unverified, hypothetical parser error messages; both now include the exact request that reproduces the documented error message.
- `registry.Tool` had no `json:"..."` tags, so `json.Marshal` emitted capitalized Go field names (`Slug`, `Name`, ŌĆ”). The new search bar's JS reads lowercase (`t.slug`, `t.name`, ŌĆ”), so every search silently returned zero results ŌĆö no thrown error, just nothing found. Caught by an end-to-end Playwright test, not a unit test or code review. Fixed with explicit lowercase `json` tags; this also fixes the same latent inconsistency in `GET /api/v1/tools`'s response shape.
- No favicon was ever defined, so every page load triggered a browser-automatic `/favicon.ico` request that 404'd (a real, if benign, console error) ŌĆö fixed.
- `README.md`'s Architecture `sequenceDiagram` Mermaid block failed to render on GitHub ("Expecting '()', ... got 'NEWLINE'") because it used unescaped `<name>`/`<slug>` placeholders in unquoted participant/message text; GitHub's Mermaid parser doesn't reliably handle HTML-entity-escaped angle brackets there. Fixed by rewording the diagram to avoid special characters entirely.
- `tool-common.js`'s `collectOptions()` only coerced `<input type="number">` values to a JSON number; any `<select>`-based numeric option (e.g. YAML Formatter's `indent`, QR Code's `size`) was sent as a JSON string, which `json.Unmarshal` into a Go `int` field rejects outright ŌĆö every web-page request using those selects silently 400'd with `invalid options object`. This had been broken since those tools were first built; only surfaced now while browser-testing the new YAML Formatter `style` select. Fixed by also coercing `<select>` values that look like a plain integer/decimal.
- The sticky header bar's content (hamburger, brand, search, theme toggle) was pinned to the far-left/far-right edges of the viewport while the page content below it was centered at `max-width: 960px` ŌĆö visually misaligned on wide screens. Fixed with a `.topnav-inner` wrapper matching the same centered width, applied once in the shared `layout.html`. The nav drawer's separate overlap-avoidance (`position: fixed` + scrim) was verified unaffected before and after, in screenshots, since it's unrelated to the header's own flex layout.
