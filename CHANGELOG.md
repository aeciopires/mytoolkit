## [1.0.0] - 2026-07-10

### Added

- Initial implementation of MyToolkit: a Go web application and CLI exposing 11 tools — JSON Tree Viewer, JSON Formatter, YAML Formatter, Password Generator, JWT Encode/Decode, QR Code Generator, Character/Word/Line Counter, URL Encode/Decode, Hash Generator, Base64 Encode/Decode, and Case Converter (added beyond the original scope).
- REST API under `/api/v1/tools/<slug>` for every tool, with a shared JSON success/error envelope.
- CLI subcommand per tool, with a shared `--in`/`--out` (file-or-stdio) convention.
- `mytoolkit --version`/`-v`, reading from the repo-root `VERSION` file — the single source of truth also used to tag Docker images.
- Server-rendered web UI with a Material Design 3–inspired dark/light theme; pages accept a `?theme=light|dark` override for deterministic screenshots/demos.
- `/healthz`, `/readyz`, `/metrics` (Prometheus), and `/api/v1/metrics/ranking` (usage ranking) endpoints.
- Structured JSON logging via zerolog, with request correlation IDs.
- Multi-stage, multi-arch (linux/amd64, linux/arm64) Dockerfile, validated with `docker buildx`, and `docker-compose.yml`.
- `make docker-push`: interactive Docker Hub publish (hidden credential prompt, `--password-stdin`, never logged).
- Helm chart under `helm/mytoolkit` — Prometheus scrape annotations, `helm-docs`-generated chart README, deployed and `helm test`-verified against a real `kind-multinodes` cluster.
- Unit tests for every tool's business logic, plus per-feature documentation (`docs/api|cli|testing`, with a Mermaid workflow diagram each) and `.skills/`.
- JSON to TOON Converter (`json-toon`), the 12th tool — converts JSON into [TOON](https://github.com/toon-format/spec) to reduce LLM token usage (tabular arrays, minimal quoting, indentation-based nesting). REST (`POST /api/v1/tools/json-toon`) and CLI (`mytoolkit json-toon`) are full Go implementations like every other tool. The web page is the first exception in this app: it converts entirely client-side via an independent JavaScript implementation (`internal/web/static/js/json-toon.js`), so no JSON is ever sent to the server from the interactive tool — verified by grepping the JS file for network APIs (none) and by an empty browser network tab during use. Introduces a new reusable `data-client-side` convention on `.tool-panel` (`registry.Tool.ClientSide`) for any future browser-only tool. Because two implementations of the same algorithm exist, `mytoolkit_tool_usage_total{tool="json-toon"}` only reflects REST/CLI usage — web conversions aren't visible to server-side metrics (same caveat as CLI usage on every other tool). The Go and JS implementations are verified against one shared fixture table (20/20 cases match exactly).
- JWT Encode/Decode: multi-algorithm support via `golang-jwt/jwt/v5` (already a dependency, previously only used for HMAC). Adds RSA (`RS256`/`RS384`/`RS512`), RSA-PSS (`PS256`/`PS384`/`PS512`), ECDSA (`ES256`/`ES384`/`ES512`), and EdDSA alongside the original HMAC algorithms — 13 total — selectable via a new web UI combobox, plus the existing CLI `--algorithm` flag and REST `options.algorithm`. `HS256` remains the default, unchanged. Asymmetric algorithms sign/verify with a new `key` parameter (PEM-encoded — private key to encode, public key to verify a decode; CLI: `--key <file|->`; REST/web: `options.key`), independent of the existing HMAC-only `secret`. Verification derives which of `secret`/`key` applies from the token's own `alg` header, not a caller hint, so pasting the wrong one fails verification cleanly (`valid: false`) instead of erroring or silently succeeding — verified directly, including a full round trip through the CLI, REST, and the actual running web page against a real `openssl`-generated RSA key pair.

- Shared navigation shell on every page: a hamburger menu opening a Material Design 3 "modal navigation drawer" listing all 12 tools (with a scrim, closable via scrim click/✕/Escape); a client-side search bar (`#tool-search`) that filters tools live by name and description with zero network calls; a "← Back to Home" button on every tool page; and a footer (developer/contact links) on every page.
- M3 component pass across the web UI, following `m3.material.io`'s spacing, buttons, icon-buttons, checkbox, lists, navigation-bar, navigation-drawer, radio-button, search, switch, and text-fields guidance: a 4dp spacing scale and state-layer opacity tokens in `theme.css`, restyled checkboxes and a new switch component (single on/off settings like "Decode" now use a switch instead of a checkbox), consistent text-field focus rings, and shared list-item styling reused by both the nav drawer and search results.
- `internal/registry/registry_test.go`: a regression test (`TestToolJSONFieldsAreLowercase`) locking in `registry.Tool`'s JSON contract.
- `internal/web/static/icons/favicon.svg`, referenced via `<link rel="icon">`.
- JSON Tree Viewer overhaul: error messages now report the exact 1-indexed line and column of the problem (e.g. `invalid character '}' looking for beginning of value (at line 4, column 1)`), not just a bare parser message; trailing content after a complete JSON value (e.g. `{"a":1}garbage`, or two concatenated values) is now rejected instead of silently ignored; the web page gained an explicit "Generate Tree View" action (conversion no longer happens on every keystroke — large pasted API responses made that wasteful and janky), plus "Expand All"/"Collapse All"/"Clear" buttons; tree nodes are now color-coded by value type (VS Code–inspired: keys, strings, numbers, booleans, and null each get a distinct, theme-aware color) and show `Object {N keys}`/`Array [N items]` summaries instead of a generic `type(size)` placeholder.
- JSON Formatter web page overhaul: now runs entirely client-side via the browser's native `JSON.parse()`/`JSON.stringify()` (no `fetch()` call from the interactive tool), so validation results and error messages exactly match real Node.js/browser behavior instead of Go's `encoding/json` wording. New, explicit actions replace live-as-you-type formatting: "Validate JSON" (green success banner on valid input, red error banner with the exact `JSON.parse()` message on invalid input, output untouched either way), "Beautify" (2-space indent), "Minify", "Copy Output", "Clear". Adds a new `.success-banner` style (light/dark) alongside the existing `.error-banner`. REST (`POST /api/v1/tools/json-format`) and CLI (`mytoolkit json-format`) are unchanged, still full Go implementations for scripted use; as with JSON to TOON Converter, `mytoolkit_tool_usage_total{tool="json-format"}` no longer reflects web page usage, only REST/CLI.
- Two new tools (13th and 14th), both powered by [`sigs.k8s.io/yaml`](https://github.com/kubernetes-sigs/yaml) — the library Kubernetes itself uses to convert between its typed API objects and YAML manifests:
  - **YAML to JSON Converter** (`yaml-to-json`) — converts a YAML document to pretty-printed JSON via `YAMLToJSONStrict`, which (unlike the library's plain `YAMLToJSON`) rejects duplicate mapping keys instead of silently keeping one in an undefined order.
  - **JSON to YAML Converter** (`json-to-yaml`) — converts a JSON document to YAML. Validates input with `encoding/json.Unmarshal` before calling into the library, because `sigs.k8s.io/yaml.JSONToYAML` parses its input as YAML internally and was found to silently accept invalid JSON (trailing commas, unquoted keys, single-quoted strings, even a bare comment line) rather than reject it.
  - Both inherit — and document, rather than hide — the library's YAML 1.1 boolean-resolution behavior (the "Norway problem"): unquoted `yes`/`no`/`y`/`n`/`on`/`off` in source YAML resolve to JSON booleans, not strings; conversely, converting a JSON string like `"NO"` to YAML automatically quotes it to avoid creating that same ambiguity.
  - This is a deliberate second YAML library choice alongside YAML Formatter's `gopkg.in/yaml.v3`: the Formatter needs `yaml.Node` to preserve comments/anchors for a same-format round trip; these converters don't have one to preserve, so `sigs.k8s.io/yaml`'s JSON-shaped decode/encode is the better fit.
  - Full REST (`POST /api/v1/tools/yaml-to-json`, `POST /api/v1/tools/json-to-yaml`), CLI (`mytoolkit yaml-to-json`, `mytoolkit json-to-yaml`), and web UI support, following the standard generic wiring.
- **Kubernetes YAML Validator** (`k8s-validate`, 15th tool) — validates that a YAML document (or a `---`-separated multi-document stream, as accepted by `kubectl apply -f`) is well-formed and has the two fields every Kubernetes API object requires: a non-empty `apiVersion` and `kind`, plus an object-shaped `metadata` if present. Each document is converted via `sigs.k8s.io/yaml.YAMLToJSONStrict` — the exact library kubectl/client-go/the API server use — so a document rejected here would also be rejected by a real cluster, for the same reason. Combines two YAML libraries deliberately: `gopkg.in/yaml.v3` splits the multi-document stream (reusing YAML Formatter's technique, since `sigs.k8s.io/yaml` has no multi-document API), then `sigs.k8s.io/yaml` validates each isolated document. A hard YAML syntax error aborts the whole request (HTTP 400); a semantic problem (missing `kind`, wrong field type) is reported per-document without stopping the rest of the batch, and the REST response is still `success: true` (the validator ran) with `data.valid: false` (the manifests aren't valid) — the same distinction a linter's exit code makes from its findings. The CLI exits `1` on any invalid document, usable directly in CI. Explicitly does **not** validate against any specific resource's full schema (Deployment, Service, a CRD, ...) — that needs the full Kubernetes OpenAPI schema database, out of scope for this lightweight tool; use `kubectl apply --dry-run=server` or `kubeconform` for that. The web page needed no custom JavaScript: `tool-common.js`'s existing `JSON.stringify(json.data, null, 2)` fallback (previously unused by any tool) renders the structured result directly.
- Character, Word & Line Counter: added a "Clear" button that empties the input and resets all four stats to `0` (previously required manually selecting and deleting the textarea contents).
- YAML Formatter overhaul, driven by a re-read of the YAML spec (yaml.org/spec/1.2.2): (1) **fixed a real spec-compliance bug** — multi-document streams (`---`-separated) previously had every document after the first silently discarded; `Format` now decodes and reformats the entire stream, and the encoder re-inserts `---` between documents automatically. (2) Added a `style` option (`block`, default, or `flow`) that forces consistent collection style across the whole document — the YAML equivalent of JSON Formatter's pretty/minify, and always lossless per the spec ("node style is a presentation detail... not reflected in the representation graph"). Previously the formatter only re-indented and left flow-style input (`{a: 1}`) untouched, so mixed-style input never actually got consistent formatting. New `--style` CLI flag and web "Style" select. A whitespace/comment-only input (which decodes to zero YAML documents without a parse error) is now treated as empty input rather than silently returning an empty result. Verified and documented actual `yaml.v3` behavior for comments, anchors/aliases, merge keys, tab-indentation rejection, and the "Norway problem" (`yes`/`no`/`NO`) — all preserved correctly or already spec-correctly rejected; see `.skills/yaml-format/SKILL.md`.
- Grafana dashboard for the `prometheus`/`grafana` `docker-compose.yml` services: `observability/mytoolkit-dashboard.json` covers every metric the app exposes — HTTP request rate/error-rate/latency percentiles by tool and status (`mytoolkit_http_requests_total`, `mytoolkit_http_request_duration_seconds`), per-tool usage (`mytoolkit_tool_usage_total`, the same data behind `GET /api/v1/metrics/ranking`), and Go runtime/process health (goroutines, memory, GC, CPU, open FDs, network I/O). Auto-provisioned via `observability/grafana/provisioning/{datasources,dashboards}/*.yml` — `docker compose up --build` now gives a working dashboard with zero manual Grafana setup. Every query was verified against a real running binary's `/metrics` output, and the whole stack was verified end-to-end (real traffic, real Grafana login, screenshotted rendering) — see `.skills/observability/SKILL.md`.

### Changed

- Go/HTML/CSS/JS source (`cmd/`, `internal/`) moved under `src/` (its own Go module root), separating application source from planning, documentation, and deployment files at the repo root.

### Fixed

- `README.md` previously listed a non-existent "SHA-1024" hash algorithm; the Hash Generator implements MD5, SHA-1, SHA-256, and SHA-512.
- The Helm chart's test hook was initially placed at `tests/test-connection.yaml`; Helm only discovers hooks under `templates/`, so it was silently never run until moved to `templates/tests/test-connection.yaml`.
- Verification pass across all 12 tools: added missing Mermaid workflow diagrams to 7 `docs/api/<tool>.md` files that didn't have one (base64, case-convert, hash-gen, jwt, qrcode, text-count, url-encode).
- `docs/api/url-encode.md` documented a nonexistent `options.mode` field; the real field is `options.decode` (boolean). Fixed to match `urlencode.Options`.
- `docs/cli/url-encode.md`'s example used `echo` (which appends a trailing newline that gets percent-encoded as a stray `%0A`); switched to `echo -n` and documented why.
- `docs/cli/json-tree.md`'s example output used a hand-typed compact JSON format that didn't match the real `json.MarshalIndent` output; corrected to the actual (fully expanded) output.
- `docs/api/json-format.md` and `docs/api/yaml-format.md` error examples used unverified, hypothetical parser error messages; both now include the exact request that reproduces the documented error message.
- `registry.Tool` had no `json:"..."` tags, so `json.Marshal` emitted capitalized Go field names (`Slug`, `Name`, …). The new search bar's JS reads lowercase (`t.slug`, `t.name`, …), so every search silently returned zero results — no thrown error, just nothing found. Caught by an end-to-end Playwright test, not a unit test or code review. Fixed with explicit lowercase `json` tags; this also fixes the same latent inconsistency in `GET /api/v1/tools`'s response shape.
- No favicon was ever defined, so every page load triggered a browser-automatic `/favicon.ico` request that 404'd (a real, if benign, console error) — fixed.
- `README.md`'s Architecture `sequenceDiagram` Mermaid block failed to render on GitHub ("Expecting '()', ... got 'NEWLINE'") because it used unescaped `<name>`/`<slug>` placeholders in unquoted participant/message text; GitHub's Mermaid parser doesn't reliably handle HTML-entity-escaped angle brackets there. Fixed by rewording the diagram to avoid special characters entirely.
- `tool-common.js`'s `collectOptions()` only coerced `<input type="number">` values to a JSON number; any `<select>`-based numeric option (e.g. YAML Formatter's `indent`, QR Code's `size`) was sent as a JSON string, which `json.Unmarshal` into a Go `int` field rejects outright — every web-page request using those selects silently 400'd with `invalid options object`. This had been broken since those tools were first built; only surfaced now while browser-testing the new YAML Formatter `style` select. Fixed by also coercing `<select>` values that look like a plain integer/decimal.
- The sticky header bar's content (hamburger, brand, search, theme toggle) was pinned to the far-left/far-right edges of the viewport while the page content below it was centered at `max-width: 960px` — visually misaligned on wide screens. Fixed with a `.topnav-inner` wrapper matching the same centered width, applied once in the shared `layout.html`. The nav drawer's separate overlap-avoidance (`position: fixed` + scrim) was verified unaffected before and after, in screenshots, since it's unrelated to the header's own flex layout.
